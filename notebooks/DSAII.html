The Big Five Notations

Notation Analogy Formal (constants) Formal (limit) Intuition
O(g) ≤ f(n) ≤ c·g(n) for some c, n ≥ n₀ lim f/g < ∞ Upper bound
Ω(g) ≥ f(n) ≥ c·g(n) for some c, n ≥ n₀ lim f/g > 0 Lower bound
Θ(g) = O(g) ∩ Ω(g) lim f/g = c, 0 < c < ∞ Tight bound
o(g) < f(n) < c·g(n) for any c lim f/g = 0 Strictly slower
ω(g) > f(n) > c·g(n) for any c lim f/g = ∞ Strictly faster

Growth Rate Hierarchy

  1  <  log n  <  √n  <  n  <  n log n  <  n²  <  n³  <  2ⁿ  <  n!  <  nⁿ
  ────────────────────────────────────────────────────────────────────────────────────►
  slower growth                                                          faster growth
Key facts: log n ∈ o(nᵅ) for any α > 0 · nᵇ ∈ o(αⁿ) for any α > 1 · Changing log base = multiplying by constant · log(n!) ∈ Θ(n log n)

Worked Example 1: Direct Proof (from 00Review slides)

📐 Slide Example — Direct Proof

Show: n log n ∈ O(n²)

Technique: Find c and n₀ such that n log n ≤ c·n²

Proof: Let c = 1 and n₀ = 1.

  Base: 1 · log(1) ≤ 1 · (1²)  →  0 ≤ 1  ✓

  For all n > 1:
      log n ≤ n           // key inequality
    → n · log n ≤ n · n
    → n log n ≤ n²       ✓  // with c=1, n₀=1

Worked Example 2: Proof by Contradiction (from 00Review slides)

📐 Slide Example — Contradiction

Show: n² ∉ O(n)

Proof: Assume n² ∈ O(n). Then ∃c such that c·n ≥ n² for all large n.

      c · n ≥ n²
      c ≥ n             // divide both sides by n

  But n is an increasing variable and c is a constant.
  Contradiction! No constant c can be ≥ all values of n.  ∎

Worked Example 4: Direct Proof for Θ (from PS1 Solution)

📝 PS1 Problem 1.1

Given: f(n) = 2n² log n + 4n, g(n) = n² log n. Show f(n) = Θ(g(n)).

Strategy: Find c and n₀ such that f(n) ≤ c·g(n).

  Choose c = 4, n₀ = 2:

      2n² log n + 3n  ≤  4n² log n
      3n  ≤  2n² log n
      3  ≤  2n log n          // divide by n

  For n ≥ 2:  2(2)(1) = 4 ≥ 3  ✓
  This grows as n grows, so holds for all n ≥ 2.  ∎

Topic 2: Breadth-First Search (BFS)

BFS(g, s):
    toVisit.enqueue(s)
    depth[s] = 0
    depth = [-1, -1, ..., -1]   // |V| times
    While toVisit not empty:
        cur = toVisit.dequeue()
        layer = depth[cur]
        for v in neighbors(cur):
            if v does not have a depth:
                depth[v] = layer + 1
                toVisit.enqueue(v)

Key theorem: A graph is bipartite if and only if it has no odd-length cycles.

Detection via BFS: Run BFS and track layers. If a non-tree edge connects two nodes at the same parity (both odd or both even layer), the graph is NOT bipartite.

Topic 4: DFS & Edge Classification

DFS Algorithm (Recursive)

DFS-VISIT(G, u):
    time = time + 1
    u.d = time              // discovery time
    u.color = GRAY
    for each v in G.Adj[u]:
        if v.color == WHITE:
            v.π = u
            DFS-VISIT(G, v)   // → Tree Edge
    u.color = BLACK
    time = time + 1
    u.f = time              // finish time

Topic 5: Topological Sort

Definition: Given a DAG, produce a linear ordering where if u→v exists, u comes before v.

Key Insight Topologically sorted vertices appear in reverse order of their DFS finish times

Modification to DFS: when a node finishes (turns BLACK), prepend it to the output list.

⚠️ Watch Out

If DFS encounters a GRAY node during topological sort, there is a cycle and no valid topological sort exists! This is the check if v.color == GRAY: // Cycle! in the modified DFS-VISIT.

Topic 6: Strongly Connected Components

Kosaraju-Sharir Algorithm

1
DFS on G
Run DFS, record all finish times
2
Transpose G → G'
Reverse all edge directions
3
DFS on G'
Process in decreasing finish time from Step 1
4
Read SCCs
Each DFS tree in Step 3 = one SCC

🚨 PS2 Problem 2 — Broken Alternative Algorithm

Question: Does running DFS again on the same G (NOT the transpose) in increasing finish time order also find SCCs?

Answer: NO! This fails. You must construct a counterexample — a graph where an SCC has two parent SCCs that are not connected to each other.

Topic 7: Universal Sink (PS2 Problem 3)

📝 PS2 Problem 3 — O(|V|) Algorithm Design

Problem: Given a directed graph as an adjacency matrix, find a universal sink (0 outgoing, |V|−1 incoming edges) in O(|V|) time.

Key Insight: Each matrix lookup eliminates one candidate

  Check G[u][v]:

  ┌─────────────────────────────────────────────────────┐
  │  If G[u][v] = 1:  u has outgoing edge              │
  │                   → u ELIMINATED (can't be sink)    │
  │                   → v might still be sink           │
  │                                                     │
  │  If G[u][v] = 0:  v missing incoming edge from u    │
  │                   → v ELIMINATED (can't be sink)    │
  │                   → u might still be sink           │
  └─────────────────────────────────────────────────────┘

Algorithm

u = 0, v = 1          // two candidates
while v < |V|:
    if G[u][v] == 0:    // v eliminated
        v = v + 1
    else:               // u eliminated
        u = v
        v = v + 1
// u is the only remaining candidate
// Verify: sweep row u (all 0s?) and column u (all 1s except u,u?)

Runtime: O(|V|) — each step eliminates one node, then one O(|V|) verification pass.

Topic 9: Solving Recurrences

Method 1: Direct Iteration (PS3)

📝 PS3 — Direct Solve: T(n) = 2T(n−1) + 1 
  T(n) = 2T(n-1) + 1
       = 2(2T(n-2) + 1) + 1  =  4T(n-2) + 3
       = 4(2T(n-3) + 1) + 3  =  8T(n-3) + 7
       ...
       = 2d T(n-d) + (2d - 1)

  Set d = n-1 to reach base case T(1):
       = 2n-1 · T(1) + 2n-1 - 1
       = 2n-1 + 2n-1 - 1
       ∈ Θ(2ⁿ)

Method 2: Substitution (Guess & Check) (PS3)

📝 PS3 — Substitution: T(n) = T(n/2) + n, guess T(n) = O(n)

Prove: T(n) ≤ cn for some c, n ≥ n₀

  T(n) = T(n/2) + n
       ≤ c(n/2) + n         // apply inductive hypothesis
       = n(c/2 + 1)
       ≤ cn                 // need c/2 + 1 ≤ c → c ≥ 2  ✓

With c = 2, the proof holds for all n ≥ 1. ∎

⚠️ Substitution Pitfall (from Slide 022)

For T(n) = 2T(n/2) + n, guessing T(n) ≤ cn FAILS:

  T(n) ≤ 2·c(n/2) + n = cn + n    // can't show ≤ cn!

You need a tighter guess: T(n) ≤ cn log n, which does work.

Worked Examples (from PS4 Solution)

📝 PS4 — Master Theorem Problems

1. T(n) = 4T(n/3) + n²

  k = log₃(4) ≈ 1.26      nᵏ = n1.26
  f(n) = n²               n² grows faster than n1.26
  → Case 3: Θ(n²)

2. T(n) = 27T(n/3) + n²

  k = log₃(27) = 3        nᵏ = n³
  f(n) = n²               n³ grows faster than n²
  → Case 1: Θ(n³)

3. T(n) = 4T(n/2) + n²

  k = log₂(4) = 2         nᵏ = n²
  f(n) = n²               nᵏ = f(n)
  → Case 2: Θ(n² log n)

4. T(n) = 3T(n/3) + n log n ⚠️ NO CASE APPLIES

  k = log₃(3) = 1         nᵏ = n
  f(n) = n log n           Differs by only a log factor, not polynomial!
  → Master Theorem does NOT apply

5. T(n) = 3T(n/4) + n log n

  k = log₄(3) ≈ 0.79      nᵏ = n0.79
  f(n) = n log n           n log n grows faster than n0.79
  → Case 3: Θ(n log n)

Topic 11: Karatsuba Multiplication

The Setup

Multiply two n-digit numbers x and y. Split each into halves:

     x = 10n/2 · A + B        y = 10n/2 · C + D

     x · y = 10n(AC) + 10n/2(AD + BC) + BD

Naive vs Karatsuba — Side by Side

Naive: 4 Multiplications

Compute AC, AD, BC, BD separately

T(n) = 4T(n/2) + 5n
k = log₂(4) = 2 → Case 1: Θ(n²)
No improvement over grade school!

Karatsuba: 3 Multiplications

Compute AC, BD, (A+B)(C+D)
Then: AD+BC = (A+B)(C+D) − AC − BD

T(n) = 3T(n/2) + 8n
k = log₂(3) ≈ 1.585 → Case 1: Θ(n1.585)
Faster!
💡 Big Takeaway

By analyzing the recurrence with the Master Theorem, you can see that eliminating even one recursive call (4→3) drops the exponent significantly. This is the core Karatsuba insight.

Topic 12: Closest Pair of Points

Algorithm Walkthrough

📍 Closest Pair — Full Visual (from Slide 024) 
Step 1: Divide at median x-coordinate

    ·1                          ·2
         ·5          ·7
    ·3          ·6         ·4
                     ·8
  ─────────── │ ───────────
  LeftPoints  │  RightPoints

Step 2: Conquer — recurse on each half

    δ_L = closest pair in left
    δ_R = closest pair in right
    δ = min(δ_L, δ_R)

Step 3: Combine — check the "runway" (strip)

               │←──── 2δ ────→│
               │   RUNWAY     │
    ·          │  ·5     ·7   │         ·
               │       ·6     │
    ·          │              │    ·
               │     ·8       │
  ─────────────┼──────────────┼──────────
               │              │

  For each point in the runway, compare to
  the next 7 points above (sorted by y).
  Why only 7? Geometric packing argument:
  at most ~8 points fit in a 2δ×δ box.
Closest Pair Recurrence T(n) = 2T(n/2) + Θ(n) → Case 2: Θ(n log n)
⚠️ Critical Optimization

Don't re-sort the runway by y each time! That gives T(n) = 2T(n/2) + Θ(n log n) → Θ(n log²n).

Instead: Each recursive call also returns points sorted by y. In the combine step, merge the two y-sorted lists (like mergesort). This keeps combine at Θ(n).

Quickselect (from Slide 025)

  [2  5  7  3  6  4  1]  8  [10  9  11  12]
  position: 0-6          7     8-11

  Looking for i=4 (4th smallest):
  Since 4 < 7 (pivot position), recurse LEFT only

  Since 7 == 7, return pivot (8) — FOUND!

ScenarioRecurrenceRuntime
Best (median pivot)S(n) = S(n/2) + nΘ(n)

Median of Medians — Guaranteeing a Good Pivot

1
Chunk
Break list into groups of 5
2
Local Medians
Find median of each group (insertion sort, constant time per group)
3
Recurse
Call Quickselect on the list of medians to find their median

This pivot is guaranteed to be within the middle 40% of the sorted list. Worst case, you recurse on 70% of the elements.

Full Runtime Derivation (from Slide 025)

📐 Quickselect + Median of Medians Runtime 
Median of Medians runtime:
  M(n) = S(n/5) + Θ(n)

Quickselect with MoM:
  S(n) ≤ S(7n/10) + M(n) + Θ(n)
       = S(7n/10) + S(n/5) + Θ(n)

Prove S(n) ≤ 10cn by induction:

  S(n) = S(n/5) + S(7n/10) + cn
       ≤ 10c(n/5) + 10c(7n/10) + cn     // inductive hypothesis
       = 2cn + 7cn + cn
       = 10cn                             // exactly what we needed!  ✓

  Therefore: S(n) = Θ(n)  — linear time!
💡 Practical Note (from Slide 025)

Median of Medians has large constants. In practice, just use a random pivot for Quicksort — it's fast (random() is cheap), and the expected runtime is Θ(n log n) because unbalanced partitions are very unlikely. If you need guaranteed Θ(n log n), use Mergesort instead.
